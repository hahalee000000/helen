> **状态**：✅ 已实现（v1.22）  
> **版本**：v1.22 (proposed)  
> **作者**：rxx + Claude  
> **日期**：2026-07-17

## 一、Executive Summary

本提案解决 Helen 上下文管理的两个核心问题：

1. **Active Context 错误共享**：当前同一 interpreter 内所有 agent 共享一个 context，违反"agent 隔离"原则
2. **Transcript 无法区分 agent**：所有消息混在同一个平面列表里，没有 `agent_name`、`invocation_id`、`parent_invocation_id` 等元数据，事后无法区分"哪些消息是 A 产生的、哪些是 B 产生的"

**核心设计原则**：

> **每次 agent `main {}` 执行都是 fresh context；main {} 退出后 context 丢弃；transcript 忠实记录一切；记忆管理是程序的事，不是语言的事。**

**核心产物**：**Invocation Tree（调用树）**——以 `invocation_id` + `parent_invocation_id` 构建的层级执行记录，让 transcript 从"平面消息列表"升级为"可查询的执行树"。配套提供 **`search_transcript(query)`** 基于内容的全文搜索，让用户根据**内容**而非 ID 找到完整上下文。

## 二、问题陈述

### 2.1 当前状态

**Active Context 全局共享**：

```python
# helen/interpreter/interpreter.py
class Interpreter:
    def __init__(self):
        self._interpreter_history = []        # ← 一个！
        self._agent_context = AgentContextManager()  # ← 一个！
        # session_id 也只有一个
```

实测结果（`/tmp/test_agent_context.helen`）：

```
Before any agent: 0 messages
A starts → 0 messages → after llm act: 2 messages
A ends → 2 messages still (❌ 没释放)
B starts → 2 messages (❌ B 看到 A 的消息)
B after llm act → 4 messages
A again → 4 messages (❌ A 的第二次运行也不是 fresh)
```

**Transcript 没有 agent 元数据**：

```json
{
    "type": "message",
    "role": "user",
    "content": "Say exactly: I am agent A",
    "uuid": "4fbd27df3a5c",
    ...
    // ❌ 没有 agent_name
    // ❌ 没有 invocation_id
    // ❌ 没有 parent_invocation_id
}
```

### 2.2 后果

1. **Agent 隔离不完整**：变量隔离了，但 LLM 上下文没有隔离
2. **Context 污染**：A 的推理输出影响 B 的推理（可能误导）
3. **Context 膨胀**：一个程序跑 10 个 agent，所有消息累加，压缩都压不住
4. **事后无法审计**：transcript 里分不清哪些消息是哪个 agent 产生的
5. **恢复粒度粗**：`restore_context(session_id)` 只能把整个 session 都恢复，不能按 agent / 按 invocation 恢复

### 2.3 违反的设计直觉

Helen 已经有"agent 隔离"：
- 变量隔离（`let` 不跨 agent）
- 工具隔离（`@sandbox` 禁止工具）
- 作用域隔离（`ReadOnlyView` 包装参数）

**LLM 上下文没有隔离，是与现有原则不一致的 bug**。

## 三、设计原则

### 3.1 核心原则

> **语言管执行，程序管记忆。**

- Helen 是编程语言，职责是提供执行原语
- "记住什么、忘记什么"是应用程序的决策，不是语言应该替用户做的
- 与 HTTP（无状态 + 应用层 session）、Unix 进程（无状态 + 文件/env）、LLM API 本身（无状态 + 调用方传历史）同哲学

### 3.2 显式优于隐式

```helen
// ❌ 隐式记忆（当前行为）
A("问题 1")
A("问题 2")  // A 隐式地"记得"问题 1 — 什么时候该记得？什么时候不该？

// ✅ 显式传递（本提案）
let ctx1 = A("问题 1")
let ctx2 = A("问题 2", context=ctx1)  // 程序员明确表达
```

### 3.3 隔离是默认，共享是显式

| 场景 | 是否共享 active context |
|---|---|
| 同一个 agent 的 `main {}` 内部多次 `llm act` | ✅ 累积（工具循环必需） |
| 同一个 agent 的两次 `main {}` 调用 | ❌ **隔离** |
| 不同 agent 的 `main {}` | ❌ **隔离** |
| `spawn A()` 并发 | ❌ **隔离** |
| 跨 `helen` 进程 | ❌ **隔离** |

### 3.4 Active Context vs Transcript 的职责分离

| | Active Context | Transcript |
|---|---|---|
| 生命周期 | 一次 `main {}` 执行 | 永久（session 级） |
| 职责 | 支撑当前推理 | 审计 / 程序显式查询 |
| 谁能读 | 当前 `main {}` 内的 LLM 调用 | 任何人 |
| main {} 退出时 | **丢弃** | **保留** |

## 四、详细设计

### 4.1 Invocation（执行单元）定义

**Invocation** 是 agent `main {}` 的一次执行。它是 context 隔离的最小边界。

```
Invocation {
    invocation_id: UUID          // 唯一标识
    agent_name: str              // 哪个 agent
    parent_invocation_id: UUID?  // 父调用（构建调用树）
    start_time: timestamp
    end_time: timestamp
    messages: [Message]          // 该 invocation 内的所有消息
}
```

**层级结构**：

```
Session
├── Invocation inv_0 (top-level main)
│   ├── messages: [...]
│   ├── Invocation inv_1 (agent A 第 1 次运行)
│   │   ├── messages: [...]
│   │   └── (可能还有嵌套的子 invocation)
│   ├── Invocation inv_2 (agent B 运行)
│   │   └── messages: [...]
│   └── Invocation inv_3 (agent A 第 2 次运行)
│       └── messages: [...]
```

### 4.2 Message 字段扩展（向后兼容）

```python
@dataclass
class Message:
    # 现有字段
    role: str
    content: str | list[dict[str, Any]]
    tool_calls: list[dict[str, Any]]
    tool_call_id: str | None
    _token_count: int

    message_type: str | None
    priority: int

    pinned: bool

    # 新增字段（v1.22）
    agent_name: str | None = field(default=None, repr=False)
    invocation_id: str = field(default="", repr=False)
    parent_invocation_id: str = field(default="", repr=False)
```

**向后兼容**：
- 旧消息这三个字段为 `None` / `""`
- JSONL 读取时缺失字段自动填默认值
- 旧程序行为不变（只是 active context 现在正确隔离了）

### 4.3 Interpreter 状态扩展

```python
class Interpreter:
    def __init__(self):
        # 现有
        self._interpreter_history: list = []
        self._agent_context = AgentContextManager()
        self._current_agent: AgentDeclNode | None = None
        
        # 新增（v1.22）
        self._current_invocation_id: str = ""
        self._invocation_stack: list[str] = []   # 支持嵌套调用
        self._invocation_index: dict[str, dict] = {}  # invocation_id → 元数据
    
    def _enter_invocation(self, agent_name: str) -> str:
        """进入新的 invocation。返回新 invocation_id。"""
        parent_id = self._current_invocation_id
        new_id = _generate_uuid()
        
        # 推栈
        self._invocation_stack.append(self._current_invocation_id)
        self._current_invocation_id = new_id
        
        # 记录元数据
        self._invocation_index[new_id] = {
            "invocation_id": new_id,
            "agent_name": agent_name,
            "parent_invocation_id": parent_id,
            "start_time": time.time(),
            "children": [],
        }
        if parent_id and parent_id in self._invocation_index:
            self._invocation_index[parent_id]["children"].append(new_id)
        
        return new_id
    
    def _exit_invocation(self):
        """退出当前 invocation。"""
        inv_id = self._current_invocation_id
        if inv_id in self._invocation_index:
            self._invocation_index[inv_id]["end_time"] = time.time()
        
        # 弹栈
        if self._invocation_stack:
            self._current_invocation_id = self._invocation_stack.pop()
        else:
            self._current_invocation_id = ""
```

### 4.4 `_call_agent` 修改

```python
def _call_agent(self, agent, args):
    # 1. 进入新 invocation
    inv_id = self._enter_invocation(agent.name)
    
    # 2. 保存 caller 的 active context
    saved_context = self._snapshot_active_context()
    
    # 3. 清空 active context（fresh！）
    self._clear_active_context()
    
    # 4. 现有逻辑：创建隔离 environment、设置参数等
    call_env = Environment()
    # ... (现有代码) ...
    
    try:
        # 执行 agent main {}
        if agent.logic is not None:
            result = agent.logic.accept(self)
            # ... (现有返回处理) ...
        # ...
    finally:
        # 5. 恢复 caller 的 active context（agent 的 context 丢弃）
        self._restore_active_context(saved_context)
        
        # 6. 退出 invocation
        self._exit_invocation()
        
        # 7. 现有清理逻辑（写回 shared let、恢复 environment 等）
        # ...
```

**关键点**：
- Agent 运行期间，active context 是 fresh 的
- Agent 的所有 `llm act` 调用都只看到自己 invocation 内的消息
- Agent 退出后，active context 恢复为 caller 的状态
- Agent 的消息已经写入了 transcript（SSOT），但**不影响 caller 的 active context**

### 5.4 内容搜索：`search_transcript` (v1.22 新增)

**动机**：一般场景下用户很难记住 invocation_id 或 session_id，但记得**内容**。需要支持"根据内容找到完整的 context"的查询模式。

**与 `search_context` 的区别**：

| | `search_context` (v1.19) | `search_transcript` (v1.22) |
|---|---|---|
| 搜索范围 | 当前 active context | Transcript（持久化） |
| 生命周期 | main {} 退出就没了 | 永久 |
| 跨 session | ❌ | ✅（`scope="all"`） |
| 跨 agent | ❌ | ✅（`agent="..."`） |
| 时间范围 | ❌ | ✅（`after` / `before`） |
| 正则 | ❌ | ✅（`regex=true`） |
| 返回粒度 | 只有匹配消息 | message / invocation / agent / session 可选 |

**API 设计**：

```helen
// 基础搜索：当前 session 内
let matches = search_transcript("认证 bug")
// [{session_id, invocation_id, agent_name, message_uuid, role, content, timestamp, snippet}, ...]

// 跨所有 session 搜索
let matches = search_transcript("认证 bug", scope="all")

// 限定 agent
let matches = search_transcript("认证 bug", agent="A")

// 限定时间范围
let matches = search_transcript("认证 bug", 
    after="2026-07-10", 
    before="2026-07-17")

// 正则匹配
let matches = search_transcript("fix.*bug", regex=true)

// 限定角色
let matches = search_transcript("TODO", role="user")

// 限制结果数
let matches = search_transcript("TODO", limit=20)
```

**`return` 参数决定返回粒度**：

```helen
// return="message" (默认) — 只返回匹配的消息本身
let matches = search_transcript("认证 bug", return="message")

// return="invocation" — 返回包含匹配消息的整个 invocation (需要 invocation_id 字段)
let invocations = search_transcript("认证 bug", return="invocation")

// return="agent" — 返回该 agent 的所有 invocation
let agent_history = search_transcript("认证 bug", return="agent")

// return="session" — 返回整个 session
let session = search_transcript("认证 bug", return="session")

// 折中：context_radius=N — 匹配消息 ± N 条消息
let matches = search_transcript("认证 bug", context_radius=3)
```

**返回格式**：

```python
[{
    "session_id": str,
    "invocation_id": str,          # v1.22 invocation tree 实现后可用
    "agent_name": str | None,      # v1.22 invocation tree 实现后可用
    "message_uuid": str,
    "role": str,
    "content": str,                # 完整内容或截断
    "snippet": str,                # 带上下文的匹配片段
    "timestamp": float,            # 如可获取
    "match_position": int,         # 匹配在 content 中的起始位置
}]
```

**使用示例**：

```helen
// 场景 1："我记得上次处理过一个认证相关的 bug"
main {
    let matches = search_transcript("认证", scope="all", limit=5)
    for m in matches {
        print("Session {m.session_id}, Agent {m.agent_name}:")
        print("  {m.snippet}")
    }
}

// 场景 2："数据库 schema 那次讨论的完整对话在哪"
main {
    let results = search_transcript("数据库 schema", return="invocation", scope="all")
    if len(results) > 0 {
        restore_context(results[0].session_id, 
                       invocation_id=results[0].invocation_id)
    }
}

// 场景 3："昨天所有关于 TODO 的讨论"
main {
    let todos = search_transcript("TODO", 
                                 role="user", 
                                 after="2026-07-16",
                                 before="2026-07-17")
    for t in todos {
        print("{t.timestamp}: {t.snippet}")
    }
}

// 场景 4：带局部上下文的搜索（轻量级）
main {
    let matches = search_transcript("fix.*bug", regex=true, context_radius=3)
    for m in matches {
        print("=== 匹配 ===")
        for ctx in m.context_before { print("  ↑ {ctx.content}") }
        print("  → {m.content}")
        for ctx in m.context_after { print("  ↓ {ctx.content}") }
    }
}
```

**实现考虑**：

- **JSONL 后端**：按行读，substring/regex 匹配，O(n)
- **SQLite 后端**：可启用 FTS5（full-text search）索引，O(log n)
- **跨 session 搜索**：遍历 `sessions/` 目录下所有 session，可能慢——建议 `scope="all"` 时提示用户
- **分阶段实现**：
  - **Phase 1**（v1.22）：基础 substring/regex 搜索，支持 `scope` / `role` / `limit`，`return="message"`
  - **Phase 2**（v1.22+，依赖 invocation tree）：支持 `agent` / `invocation_id` 过滤，`return="invocation"`
  - **Phase 3**（v1.23）：`context_radius`、SQLite FTS5 索引、时间范围过滤

**中文别名**：`搜索会话`、`搜索 transcript`。

## 七、实现路径

### Step 1: Message 字段扩展（向后兼容）

## 八、示例场景

### 8.1 典型程序的 transcript 结构

程序：
```helen
agent Researcher {
    main {
        llm act "研究 X"
        llm act "深入 Y"
    }
}

agent Writer {
    main {
        llm act "基于研究写文章"
    }
}

main {
    Researcher()   // inv_1
    Writer()        // inv_2
    Researcher()    // inv_3
}
```

Transcript：
```
inv_0 (top-level main)
├── [top message]: "开始"
├── inv_1 (Researcher 第 1 次)
│   ├── user: "研究 X"
│   ├── assistant: "研究结果..."
│   ├── user: "深入 Y"
│   └── assistant: "深入结果..."
├── inv_2 (Writer)
│   ├── user: "基于研究写文章"
│   └── assistant: "文章..."
└── inv_3 (Researcher 第 2 次)
    ├── user: "研究 X"
    └── assistant: "研究结果... (fresh!)"
```

**关键**：
- inv_3 是 fresh 的——看不到 inv_1 的 4 条消息
- inv_2 看不到 inv_1 的消息
- 但 transcript 完整记录了所有调用

### 8.2 调试：打印调用树

```helen
main {
    A()
    B()
    A()
    
    let tree = get_invocation_tree()
    print_tree(tree)
}
// 输出:
// inv_0 (top) [3 msgs]
// ├── inv_1 (A) [4 msgs]
// ├── inv_2 (B) [2 msgs]
// └── inv_3 (A) [2 msgs]
```

### 8.3 显式传递 context（程序控制）

```helen
agent A {
    params { prior_context: dict? }
    main {
        if prior_context != null {
            import_context(prior_context)
        }
        // ... 工作 ...
        return export_context()  // 返回当前 context 供下次使用
    }
}

main {
    // 第一次调用
    let ctx1 = A(prior_context=null)
    
    // 第二次调用，显式传入上次的 context
    let ctx2 = A(prior_context=ctx1.context)
    
    // 或者不传（让 A 自己 fresh 开始）
    let ctx3 = A(prior_context=null)
}
```

## 九、边界情况与开放问题

### 9.1 嵌套 agent 调用

```helen
agent Inner {
    main { llm act "inner" }
}

agent Outer {
    main {
        llm act "outer start"
        Inner()   // 嵌套调用
        llm act "outer end"   // ← 这里能看到 Inner 的消息吗？
    }
}
```

**答案**：**不能**。Inner 的 invocation 是 fresh 的，Outer 的 active context 也不包含 Inner 的消息。Outer 调用 Inner 后，恢复 Outer 的 context（调用 Inner 之前的状态）。

如果 Outer 需要 Inner 的结果，应该通过：
- 返回值：`let result = Inner()`
- 共享 store：`shared store SharedResult { ... }`
- Channel：`let ch = spawn Inner(); let result = ch.receive()`

### 9.2 REPL 体验

REPL 中每次命令都是 fresh invocation：

```
> A("问题 1")   // inv_1
> A("问题 2")   // inv_2 (fresh, 看不到问题 1)
```

如果用户想要多轮对话，**REPL 程序自己**需要显式管理：

```python
# REPL 实现（Python）
def repl():
    last_invocation = None
    while True:
        cmd = input("> ")
        if last_invocation:
            # REPL 决定：把上次 invocation 的 context 注入这次
            restore_context(session_id, invocation_id=last_invocation)
        run_helen(cmd)
        last_invocation = get_latest_invocation_id()
```

不同 REPL 实现可以有不同的记忆策略——这是 REPL 程序的自由。

### 9.3 spawn 并发

```helen
main {
    let ch1 = spawn A()
    let ch2 = spawn B()
    let r1 = ch1.receive()
    let r2 = ch2.receive()
}
```

**每个 spawn 出的 agent 是独立 invocation**，parent 都是顶层 main。它们各自有 fresh context，互不影响。

### 9.4 跨进程恢复

```helen
// 程序 1
main {
    A()
    B()
    // session_1 的 transcript 持久化到磁盘
}

// 程序 2（新进程）
main {
    // 显式恢复 program 1 的某次 invocation
    restore_context("session_1", invocation_id="inv_2")
    C()  // C 能看到 program 1 里 B 的所有消息
}
```

### 9.5 顶层 `llm act`（不在 agent 里）

程序里直接在 main {} 里调用 `llm act`，这些消息属于顶层 invocation（`agent_name=null`）。

### 9.6 与现有 `resume_session` 的关系

- `resume_session(session_id)`：替换 TranscriptStore 引用（审计层操作）
- `restore_context(session_id)`：恢复 active context（应用层操作）

两者可以组合使用。例如：
```helen
resume_session("old_session")    // 切换 transcript 到旧 session
restore_context("old_session")   // 同时恢复 active context
```

### 9.7 性能考虑

- `_invocation_index` 是内存字典，长会话可能变大
- 可以考虑 LRU 淘汰旧的 invocation 元数据（transcript 仍持久化）
- 调用树查询应该是 O(1)（用 dict 索引）或 O(log n)

### 9.8 与 BoundaryMarker 的关系

BoundaryMarker 标记压缩事件，Invocation 标记执行边界。两者正交：
- 一个 Invocation 内可以有多个 BoundaryMarker（压缩了多次）
- BoundaryMarker 也应该带上 `invocation_id`，便于查询"哪次调用的压缩历史"

## 十、迁移与向后兼容

### 10.1 旧 Transcript 文件

- 旧 JSONL 文件读取时，缺失字段自动填 `None` / `""`
- 旧消息的 `agent_name` 显示为 `"<unknown>"` 或 `null`
- 不影响功能，但无法做精细过滤

### 10.2 旧程序行为

- 现有程序继续运行
- 但现在 agent 之间 context 隔离了——这是**语义变化**
- 如果旧程序依赖 agent 共享 context（错误用法），会"看起来不同"
- 文档里明确说明这个语义变化

### 10.3 版本号

- 建议作为 **v1.22** 的主要特性
- 在 changelog 里明确标注：**破坏性变更**：agent 调用之间 context 不再共享

## 十二、待讨论

1. **嵌套调用的 active context 边界**：当前设计是"调用子 agent 后恢复 caller 的 pre-call context"。是否有场景需要"调用子 agent 后，caller 能看到子 agent 的消息"？如果有，可以提供 opt-in 的 `merge_child_context()` 函数

2. **顶层 main 的 agent_name**：`null` vs `"__top_level__"` vs `"<main>"`？

3. **invocation 元数据的持久化**：`_invocation_index` 是内存字典，是否需要持久化到单独的 metadata 文件？

4. **`list_invocations` 的分页**：长会话可能有成千上万个 invocation，API 是否需要分页？

5. **与 observability.call_stack 的关系**：是否可以复用 call_stack 的栈结构？
